# Data Governance Policy — RetainIQ Pipeline (MSBA 305)

## 1. Data Classification

| Data element | Classification | Justification |
|---|---|---|
| `phone_number` (XML) | **PII — Sensitive** | Direct identifier; hashed with SHA-256 before storage |
| `full_name` (JSON profiles) | **PII — Confidential** | Direct identifier; synthetic in this dataset but treated as PII |
| `date_of_birth`, `age` (JSON) | **PII — Confidential** | Quasi-identifier; combined with other fields enables re-identification |
| `wallet_id` | **Confidential** | Pseudonymous identifier linking all three sources |
| `amount_ngn`, `net_amount` | **Confidential** | Financial transaction values |
| `fraud_flag`, `churn_30d` | **Confidential** | Derived risk signals; sensitive for regulatory purposes |
| `transaction_type`, `channel`, `kyc_tier` | **Internal** | Operational metadata; not directly sensitive |
| Aggregated query results (Q1–Q5) | **Internal / Publishable** | No individual-level data; safe for dashboards |

---

## 2. Access Control

Three roles are defined. In Neon Postgres these map to database roles; in MongoDB Atlas to built-in roles.

| Role | Postgres | MongoDB | Scope |
|---|---|---|---|
| `etl_loader` | INSERT, TRUNCATE on fact/dim tables | `readWrite` on `retainiq` | Pipeline scripts only — no SELECT on PII columns |
| `analyst` | SELECT on all tables except raw PII columns | `read` on `retainiq` | Notebook queries, reporting |
| `admin` | ALL PRIVILEGES | `atlasAdmin` | Schema changes, index management |

**PII column restriction:** The `analyst` role should be granted SELECT on a view that excludes `full_name`, `date_of_birth`, and the raw (unhashed) `phone_number`. In the current academic implementation, a single admin credential is used; production deployment would enforce role separation.

## 3. Data Retention & Lifecycle

| Data tier | Retention period | Justification |
|---|---|---|
| Raw parquet (Google Drive) | 7 years | Nigerian Central Bank BOFIA 2020 §22 — financial records must be retained for minimum 7 years |
| Raw XML (Google Drive) | 7 years | Same BOFIA requirement applies to cross-border settlement records |
| JSON profiles (MongoDB Atlas) | 7 years | NDPR (Nigeria Data Protection Regulation) — personal data retained as long as the account relationship exists, minimum 7 years for financial records |
| Postgres analytical tables | 7 years | Mirrors source retention |
| Processed intermediate files (`data/processed/`) | Reproducible on demand — can be deleted and rebuilt | Derived artifacts; regenerated by running the pipeline |
| SHA-256 hashed phone numbers | Indefinitely | Hashed values are irreversible pseudonyms — no PII risk; useful for long-term fraud linkage |

**Deletion procedure:** Account closure triggers a GDPR/NDPR-style right-to-erasure workflow: delete the Mongo document, anonymise the Postgres rows (replace wallet_id with a tombstone hash), and log the deletion event.

## 4. Backup & Disaster Recovery

| System | Backup mechanism | RPO | RTO |
|---|---|---|---|
| Neon Postgres | Built-in point-in-time recovery (PITR); free tier supports 6-hour recovery window | 6 hours | < 1 hour |
| MongoDB Atlas | Automated cloud backups; M0 free tier takes daily snapshots | 24 hours | < 2 hours |
| Raw data files | Google Drive with version history; shared folder with team and professor | Real-time sync | Immediate |
| Code & notebooks | GitHub repository | Per-commit | Immediate |

## 5. Cloud vs. Local Trade-off Analysis

See `docs/decisions/06_cloud_vs_local.md` for the full decision record. Summary:
- **Cloud chosen for databases** (Neon + Atlas) to enable professor access without local DB installation.
- **Raw files on Google Drive** — accessible to all team members and the professor; Drive handles access control via link sharing.
- **Code on GitHub** — version-controlled, diff-visible, professor can clone and run.
- **Local processing only** — pandas transformations run on the developer's machine; no cloud compute required.

## 6. Ethical Considerations & Bias Documentation

- All three sources contain **synthetic data**. Fraud and churn signals were generated by a probabilistic model, not observed from real customer behaviour. Any production deployment must validate all models and thresholds on real data before use.
- The customer profile synthetic generator produces **near-equal gender splits and uniform age distributions** — not representative of real Nigerian mobile-money demographics (which skew younger and male per GSMA 2024 data). This limits the generalisability of demographic-segmented churn analyses.
- The XML source originates from an **Indonesian fintech** (Bilpay) repositioned as a cross-border partner feed. Currency (IDR) is not converted to NGN; cross-source financial comparisons require an exchange-rate layer not implemented in this pipeline scope.
- **SHA-256 hashing of phone numbers** is applied consistently across all sources containing phone numbers (XML only; JSON profiles do not include phone numbers). The hash is deterministic — the same phone number always produces the same hash — enabling cross-source linkage without exposing the raw number.
